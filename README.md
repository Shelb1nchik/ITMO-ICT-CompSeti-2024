Команда SELECT pg_is_in_recovery(); используется для проверки, находится ли база данных PostgreSQL в режиме восстановления.

*Значение pg_is_in_recovery:*

- f (false) означает, что база данных не находится в режиме восстановления и функционирует как основная база данных.

- t (true) означало бы, что база данных находится в режиме восстановления (например, как реплика).

## Ответы на вопросы:

***Порты 8008 и 5432 вынесены в разные директивы, expose и ports. По сути, если записать 8008 в ports, то он тоже станет exposed. В 
чем разница?***

EXPOSE:

Присутствие директивы EXPOSE в Dockerfile указывает, какие порты внутри контейнера будут "открыты" для других контейнеров внутри той же Docker-сети.

EXPOSE-порты не автоматически становятся доступными наружу, под хост (т.е., для внешних сетей).

PORTS:

Присутствие директивы PORTS в Dockerfile предназначено для привязки портов, открытых внутри контейнера, к портам хоста (т.е. вашего компьютера).

Порты, указанные в PORTS, будут доступны не только внутри сети, но и во внешние: они будут связываться с соответствующими портами хоста.

Итак, в первую очередь, "место использования" портов: путь к ним, сделанный через EXPOSE, будет открыт другим контейнерам в рамках Docker-сети. При использовании же директивы PORTS порты становятся доступными и "снаружи". 

***При обычном перезапуске композ-проекта, будет ли сбилден заново образ? А если предварительно отредактировать файлы postgresX.yml? А если содержимое самого Dockerfile? Почему?***

При обычном перезапуске композ-проекта: Выполнив команду docker-compose up, Docker Compose использует уже существующие образы и контейнеры, если они есть. Проще говоря, при простом перезапуске проекта, существующие образы не пересобираются, если не произошло изменений в конфигурациях.

Если предварительно отредактировать файлы postgresX.yml: Настройка файлов postgres0.yml и postgres1.yml не повлечет за собой пересборку образов, потому что эти файлы служат конфигурационными для Patroni и не влияют на процесс сборки Docker-образов. Docker Compose просто перезапустит контейнеры, но уже с новыми конфигами.

Если изменится сам Dockerfile: Внося изменения в содержание Dockerfile, вы влияете на процесс построения образов. Если файл Dockerfile изменен, то образ нужно пересобрать вручную с помощью команды:

```docker-compose up --build```


Почему так: Перезапуск композ проекта: Docker Compose использует кэш образов и контейнеров для ускорения работы. И просто использует уже существующие образы, если не указана явно пересборка.

Изменение файлов postgresX.yml: Эти файлы служат конфигурационными для Patroni и не участвуют в процессе сборки Docker образов. И изменения в этих файлах просто применяются при следующем запуске контейнеров.

Изменение Dockerfile: Файл Dockerfile определяет процесс сборки образа. Любые внесенные изменения в Dockerfile, потребуют пересборки образа, чтобы Docker Compose узнал о новых инструкциях и обновлениях.
